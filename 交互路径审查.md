# 交互路径审查（兽兽）

> 每次发版前，对所有可交互元素进行"点一遍"测试。
> 目的：确保每个点击都产生可预期的、即时的反馈，不漏掉静默失败的交互。

## 审查流程

### 1. 走查路线

按用户实际使用路径顺序，依次测试：

```
首页 → 单词墙 → 点击单词 → 收藏/标记/朗读/从本词学习
     → 背语法 → 语法播放 → 导航/暂停/停止
     → 复习 → 选择词单 → 学习 → 评分
     → 生词本 → 浏览/移除
     → 自动播放 → 设置选项 → 开始播放 → 导航‹› / 暂停/停止 / 收藏/标记
     → 闯关 → 开始 → 答题 → 评分
     → 真题模拟 → 选择 → 开始 → 提交
     → 错题本 → 浏览
     → 学习计划 → 新建 → 编辑 → 删除
```

### 2. 审查清单

每个页面过以下检查：

| # | 检查项 | 通过条件 |
|---|--------|---------|
| 1 | 可见按钮可点击 | 点击后触发对应函数，控制台无报错 |
| 2 | 点击反馈即时 | DOM 更新在 500ms 内完成（切词、换页等） |
| 3 | 导航边界正确 | 第一项时 prev 不崩；最后一项时 next 不崩 |
| 4 | 状态回显正确 | 收藏/标记状态在页面切换后一致 |
| 5 | 无静默失败 | 点击后没有"看似没反应"的情况 |

### 3. 如果测出问题

1. 截图+录控制台报错（如果可见）
2. 记入 `G:\reports\YYYY-MM-DD_交互审查报告.html`
3. 修复后重新跑一次相关路径

### 4. 责任

- 兽兽（我）负责每次发版前的交互路径审查
- 纳入系统测试的前置步骤
- 如果发版后发现交互问题，第一个追责的是我

### 5. 审查记录

每次审查结果写入 `G:\reports\`，格式：
```
日期 | 版本 | 审查路径数 | 发现问题 | 状态
```
